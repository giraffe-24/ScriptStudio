ScriptStudio の全体像 — 5つの役割

5つの役割が、1つの画面で地続きにつながっています。この資料が開くのは、そのうちの 「保存・管理」だけ。残り4つは「こんなこともできる」という地図として眺めてください。

企画立案

テーマ調査・市場分析でネタと根拠を洗い出し、「設計思想の6本柱」で軸を固めてから企画書に落とす。

競合チャンネルを登録して傾向を把握し、ブレない差別化の材料にする。

台本作成

自分の声で執筆。気になる箇所だけ選んで前後の文脈ごと書き直す部分再生成で、トークンも節約。

ドキュメント感覚の使い勝手——並べ替え・字数確認に加え、版を残す履歴管理（保存／履歴／復元）。この資料が掘り下げるのはここ。

SECTION 1

ツールの画面（キャプチャ）

まず入口のログイン画面、そのあとの操作は大きく 3画面。エディタのツールバーから「記録」と「履歴」を開く。 下の枠はスクリーンショットを貼るための領域（実データは後から貼る）。

ログイン画面（入口）

SCREENSHOT

スクリーンショットを貼る領域

ScriptStudio ロゴ、ユーザー名・パスワードの入力欄、「ログイン」ボタンが写ったログイン画面。

認証方式の選定 ── なぜ Google 認証にしなかったか

当初は Google 認証（OAuth 認証） を入れようとした。ただし利用者は基本的に本人。 OAuth は同意画面の審査・連携設定・運用の手間まで抱える割に、この規模では見合わない。 費用対効果を踏まえ、ID＋パスワードの自前認証（ログイン成功でサーバーがセッションを発行し、Cookie で保持）を採用した。小さく閉じた利用なら、これで必要十分。

ユーザーがやることは「書く → 記録を押す → 必要なら履歴から戻す」だけ。 保存先の Supabase は画面には出てこない（裏側で処理される）。使う人は仕組みを意識しなくていい。

SECTION 2

保存しているデータ

キホンの単位は スナップショット。 「記録」を1回押すと、そのときの内容を丸ごと固めた1件が積み上がる。台本と企画書で別々に履歴を持つ。

スナップショット1件に入っているもの

保存する内容 意味・使いみち
記録した人 ログイン名から自動で入る
記録した日時 履歴を新しい順に並べる基準
変更の要約 「何を変えたか」を1〜2文で。AI が差分から自動生成
本文の全文 この版へそっくり戻すための原本
変更量 追加・削除の行数。どれだけ変わったかの目安（台本のみ）
どの回のものか 動画の回ごとに履歴を分ける

※ 変更のあった部分「だけ」ではなく、毎回本文をまるごと保存している。これが「この版にそっくり戻す」を成立させるカギ。

2系統のスナップショット（対称設計）

台本スナップショット

table: script_snapshots

  • content = 台本の全文テキスト
  • 追加/削除の行数（diff_stats）も保持

企画書スナップショット

table: plan_snapshots

  • content = 企画書の plan.json（構造ごと）
  • 差分・要約は「整形テキスト」に変換して表示

企画書の「整形テキスト」に含む項目

タイトル 想定視聴者 視聴者の悩み 提供する価値 コンテンツの核 構成 競合との差別化 想定尺 ゴール

狙い：保存は復元しやすい JSON、表示は人が差分を追える整形テキスト。「保存の形」と「見せる形」を分けている。

SECTION 3

保存先と、それを選んだ理由

履歴は Supabase（クラウド DB） に一元保存する。なぜファイルではなく外部 DB なのか、そして実際にどんな形でデータを持っているかを見ていく。

保存の流れ

保存ボタン

画面で「記録」する

API

/api/script-versions

Supabase に1件追加

script_snapshots

1回の保存 ＝ テーブルに1行 INSERT。台本は script_snapshots、企画書は plan_snapshots に同じ仕組みで貯まる。呼び出す側（画面）は API を叩くだけ。

Supabase（Postgres）に保存

なぜ外部 DB が必要？

土台は Vercel のサーバーレス。書き込んだファイルは処理が終わると消える（揮発する）。だから履歴のような「残さないと意味がないデータ」は外部に置くしかない。

なぜ Supabase を選んだ？

  • マネージドの Postgres で運用が軽い
  • 無料枠で始められる
  • サーバー用キーで直接読み書きでき、SQL で表を管理できる

データの持ち方（テーブル設計）

1回の保存 ＝ 1行

台本の全文をまるごと残し、著者・保存メモ・前版との差分・保存時刻を添える。だから「その版に丸ごと戻す」ができる。

script_snapshots の主な列

  • content台本の全文（まるごと）
  • summary保存メモ
  • diff_stats前版との差分サマリ
  • author_name保存した人
  • created_at保存時刻

取り出すときはその回のぶんを新しい順に並べる → それが履歴。企画書も同じ形で plan_snapshots に貯まる。

SECTION 4

工夫したポイント

「消えない・止まらない・原因がわかる」を成立させるための5つの仕掛け。

1

変更の差分を保存時に記録

保存のたびに前版との差分（追加◯行／削除◯行）を計算し、diff_stats にひも付けて一緒に残す。履歴一覧はこれを見るだけで、開かなくても「どれだけ変わった版か」が一目でわかる。

2

変更内容を AI が要約（失敗しても止めない）

記録時に Claude が差分を読み、1〜2文の要約を自動生成。API が落ちても「追加◯行／削除◯行」から定型文にフォールバックし、初稿・無変更も専用の文言で必ず何かを返す。

3

7日休止を防ぐキープアライブ

Supabase の無料枠は約7日 DB を触らないと自動休止する。そこで Vercel Cron が1日1回、専用テーブルへ upsert（＝本物の DB クエリ）を送って起こしておく。Storage 読み取りも併用して二重に保険をかける。

4

保存先の健全性を診断する API

「通信に失敗」がユーザーの回線のせいか、Supabase が止まっているせいかを、推測でなく事実で切り分ける /api/health。到達不能／タイムアウト／応答ありに分類し、キーや URL などの秘密は返さない。

5

企画書は「保存＝JSON／表示＝テキスト」を分離

復元は構造を保った JSON、差分表示は人が読める整形テキスト。1つのデータに2つの見せ方を用意し、「そっくり戻せる」と「変更が追える」を両立させた。

SECTION 5

苦戦したポイント

実際に踏んだ3つの落とし穴と、そこから決めた対策。

① AI 要約の失敗が、保存まで道連れにした

症状：記録のたびに変更点を AI で1〜2文に要約していたが、AI 側がタイムアウト・失敗すると保存そのものが巻き添えで失敗。おまけ機能のはずが、履歴の生命線を止めていた。

対策：要約は「あれば嬉しいおまけ」と割り切り、失敗しても差分数（＋◯／−◯）から定型文へフォールバック。要約が無くても保存は必ず通すようにした。

② Supabase 無料枠の7日自動休止

症状：DB アクティビティが約7日ないと自動で一時停止。さらに放置すると削除され、一度死ぬとキープアライブでも復活しない（ホスト名が引けなくなる）。

対策：あくまで予防としてキープアライブ Cron を用意。停止済みはダッシュボードから復元／再作成が必要で、恒久的に止めない保証は有料プランのみ、と割り切って明文化した。

③ 失敗の原因が切り分けられない

症状：「保存に失敗」とだけ出て、回線なのかサーバーなのか、設定ミスなのか分からない。

対策：診断 API で事実ベースに分類。あわせて、サーバー用キーの取り違え（publishable キーを誤設定）も設定チェックで弾き、ヒントを返すようにした。

発展課題（任意）

AIが動く機能

基本の課題（画面〜苦戦）に加えて、アプリの随所に 「AIが動く機能」 を組み込んだ。共通するのは、全部を渡さず「対象だけ」を AI に渡すという設計。3つの例で示す。

変更を AI が要約

渡す：前の版との差分 → 返る：1〜2文の要約

記録するたび、履歴に「何を変えたか」が自動で並ぶ。

台本の部分再生成

渡す：選んだ箇所＋前後（各〜1000字）→ 返る：その範囲だけ書き直し

執筆ペインで気になる箇所を選ぶだけ。全文を作り直さないので他がブレず、トークンも節約。

企画書の部分修正

渡す：直したい項目だけ → 返る：その項目だけ更新

想定視聴者・悩みなど、項目単位で AI に相談。企画書ぜんぶは触らない。

道を外させない工夫（ハーネス）

どれも対象を絞って渡すのが肝。範囲が狭いほど、AI は暴走せず・他をブレさせず・トークンも軽い。さらに要約は、API が落ちても「追加◯行／削除◯行」の定型文にフォールバックして本体（保存）は止めない。「どの作業を AI に任せ、どうすれば道を外さず動くか」を設計した部分。

SUMMARY

まとめ

この保存基盤が担保している3つの価値。

1

いつでも戻せる

1記録＝1スナップショットで、誰が・いつ・何を・全文を保持。過去の版へそっくり復元できる。

2

変更が一目で追える

保存のたびに前版との差分（＋◯／−◯）を一緒に記録。履歴一覧を見れば、開かずに「どれだけ変わった版か」がわかる。

3

壊れにくく、原因がわかる

キープアライブ・保存先診断・要約フォールバックで、「通信に失敗」を予防し、起きても切り分けられる。

要は「データは全文で残す・変更は差分で追える・落ちる前提で守りを固める」。この3点が、履歴を安心して使える土台になっている。
