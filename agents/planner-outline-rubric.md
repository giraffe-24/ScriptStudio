# 目次案チェックリスト（Planner / Architect 用）

> ポリシー SSOT：[config/planning.md](../config/planning.md)
> `/企画` は [planner.md](planner.md)、`/構成` は [architect.md](architect.md)

---

## 生成前（プロンプト設計）

- [ ] JSON 例に「本題 -」「まとめ」「導入（0:00）」を含めていない
- [ ] 目次案は内容見出しの例だけを示している
- [ ] 尺・時間は estimatedLength にのみ書くよう指示している

---

## 生成後（自己チェック）

各 outline.section について：

- [ ] 構成ラベル（本題・まとめ・導入・付録・注意点等）単体ではない
- [ ] 「本題 -」「導入：」等の接頭辞がない
- [ ] 【】や [] で囲んだメタラベルがない
- [ ] 0:00、5分、（3:00〜）等の時間表記がない
- [ ] STEP1、1. だけの番号見出しになっていない
- [ ] 見出しだけで章の内容がわかる
- [ ] 台本 ## にそのまま出して自然

---

## 修正方針

| 検出 | 修正 |
|------|------|
| `本題 - 〇〇` | `〇〇` だけ残す |
| `まとめ` | 内容を表す見出しに置換（例：3か月続けた振り返り） |
| `導入（0:00）` | 時間を除き、内容見出しに（例：なぜ今この設定が必要か） |
| 見出しが空 | outline.content の先頭から要約して見出し化 |

---

## Architect 向け補足

構成エージェントは「導入・本題・まとめ」を内部設計の枠として使ってよい。
ただし、最終的な目次案（outline.section / 構成案のセクション見出し）にそのラベルを出さない。

内部メモ例：

```
[内部] 導入相当 → 公開見出し「共有できないとき最初に疑う3か所」
[内部] 本題相当 → 公開見出し「Geminiに食事ログを送る習慣づくり」
[内部] まとめ相当 → 公開見出し「3か月続けたときの振り返り方」
```

## Writer 向け参照

台本執筆時は planning.md §4 を厳守。outline.section を ## 見出しとして一字一句使う。
