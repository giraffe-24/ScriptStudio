# 品質基準・NGパターン（SSOT）

> このファイルは品質判定の単一の真実の源（SSOT）です。
> レビューエージェントはこのファイルに基づいて品質チェックを行います。

---

## 絶対NGパターン

以下が含まれている場合、即座に修正を求める：

| NG | 理由 | 検出パターン | 代替表現 |
|---|------|-------------|---------|
| Markdown太字 `**` | 体裁統一・貼り付け先で崩れやすい | 連続する `**` で囲んだ箇所 | 見出し・箇条書き・「」で強調 |
| 競合批判 | 信頼を損なう | 「〜より優れている」「〜はダメ」 | 「〜も良いですが、今回は〜」 |
| 誇張表現 | 40-60代に響かない | 「神」「やばい」「最強」「爆速」 | 「かなり便利」「おすすめ」 |
| 不安煽り | ブランドに合わない | 「大変なことに」「危険」「知らないと損」 | 「備えておくと安心」 |
| マウント | 距離が生まれる | 「私はできる」「当然」「常識」 | 「私みたいなズボラでも」 |
| 専門用語無説明 | 離脱につながる | API, UI, UX, クラウド等 | 必ず噛み砕いて説明 |

## 製品・アプリの事実（誤情報・未確認断定の禁止）

台本では、確認していない状態でアプリの有無・入手方法・画面上の名前を断定しない。OS やブランド・アプリの構成は変更されやすいため、ツール依存の回では次を守る。

1. 執筆前：Microsoft / Google / Apple など公式ヘルプ、または開発元の現在の製品ページを参照する（検索結果の要約だけで済ませず、本文の主張どおりか本文を自分で読む）。
2. 本文では：変更されうる文言（ボタンの位置・アプリの版）は「画面上で確認して」「自分の環境のアプリでは」と逃がす余地を残す。
3. 典型例（Microsoft Sticky Notes × Android／再発防止のため SSOT に明記）  
   Android で「Sticky Notes と名のついただけの単体アプリ」を入れるだけで Windows の付箋一覧に触れると断定するような書き方は、視聴者を誤解させうる。Microsoft の公式説明では、Android は主に Microsoft OneNote アプリの中にある付箋（Sticky Notes）から利用する。また Microsoft Launcher でフィード経由になる、という案内がある。収録では、OneNote 内で「Sticky Notes／付箋」が自分の画面上で実際どう見えるか、実機か最新公式の説明画像で確認する。  
   一次ソース（執筆時点で内容を自分で確認する）：  
   - Microsoft サポート「See your sticky notes on other devices and the web」  
     https://support.microsoft.com/office/see-your-sticky-notes-on-other-devices-and-the-web-cf4bacd0-c042-46fd-9077-ca8c82dc0236  
   - トラブル時の複数アプリへの言及例（Android で OneNote / Office / Launcher 等への言及）  
     https://support.microsoft.com/office/troubleshoot-sticky-notes-89b1bb37-ef52-4e56-a066-418d7ea0f112  
   - 複数デバイスで見られないときの説明（「OneNote が入っていて同じアカウント」等）の例  
     https://support.microsoft.com/office/where-did-my-sticky-notes-go-170fcaa4-99c1-4c21-98ff-83c8f97017f7  

チェック側（reviewer）は、上記のような製品説明としての誤りを検出したら NG に近い重大指摘として扱う（詳細は [agents/reviewer.md](../agents/reviewer.md)）。

## 推敲比較（キャリブレーション）時の絶対ルール

以下は SSOT として、推敲比較（AI原稿と手元の確定稿の読み比べ）を行うあらゆるエージェント・チャット指示に準拠する。

1. 最高の精度改善のために、文脈が足りないと判断したときは、推測で埋めず必ず利用者に質問する（例: 尺、想定視聴層、今回の企画の禁則、手直しの意図が複数ある場合の優先順位、ブランド方針とのトレードオフ）。質問を省略してはならない。
2. 手直し側を「真似する」主な観点は次の三つに置く: 言葉づかい、リズム、構成。
3. 記事（台本）によっては、AI原稿で削ったが、手元の確定稿では残している内容がある。それを「残すべきだった／削りすぎ」として拾い、次回執筆に活かす提案をする（事実と推論を混同しない）。
4. 改善提案は `config/voice.md` に閉じない。差分の本質に応じて `config/brand.md`・`config/audience.md`・`config/calibration.md`（マーカー運用）・`templates/structure-*.md`・`agents/writer.md`・`agents/architect.md`・`agents/planner.md`・`agents/reviewer.md`・`agents/titler.md` など、該当する SSOT またはエージェント指示のどれをどう変えると再現性が上がるかを指名する。リポジトリへの自動マージは行わない（人間の承認と編集後に反映する）。

推敲比較のたびに必ず上記1を満たすこと。成果として `outputs/` にレポート用 Markdown を保存することは必須としない（チャット上の提案と SSOT への人間承認後の追記で足りる）。

マーカー境界の SSOT は [config/calibration.md](calibration.md)。推敲手順は [agents/calibrator.md](../agents/calibrator.md)。

## 品質チェックリスト

### 導入部チェック（10項目）

```
□ 1. 自己紹介が正確か
     「『楽』を極めて最大の結果を出す、効率化オタクのあらきりです」

□ 2. フックが40-60代に響く表現か
     - 問いかけ型 or 共感型 or 挑発型（適度）

□ 3. 視聴者の課題/不満を言語化しているか
     - 「〜が面倒」「〜で困っている」

□ 4. 解決策を明示しているか
     - 「〜を使えば」「でも安心してください」

□ 5. 価値リストがあるか（3-8個）
     - 動画を見る理由を複数提示

□ 6. 宣言があるか
     - 「今回は〜を紹介します」

□ 7. 「自分にもできそう」と感じさせているか
     - 「ズボラでも」「簡単に」

□ 8. 不安を煽っていないか
     - NGパターンに該当しないか

□ 9. 誇張表現がないか
     - NGパターンに該当しないか

□ 10. 30秒以内に読める長さか
      - 約300-400字
```

### 本題チェック（12項目）

```
□ 1. セクション見出しが明確か
     - STEP形式 or 番号形式

□ 2. 「現在地」がわかる構成か
     - 「STEP1」「1.」などの番号

□ 3. 各セクションが3層構造か
     - 結論→手順→補足

□ 4. 結論が最初にあるか
     - 「〜は、〜ができる機能です」

□ 5. 操作が「〜をタップ」レベルまで具体化されているか
     - 抽象的な「設定を変更する」はNG

□ 6. 「なぜ」の説明が適度に入っているか
     - 手順だけでなく理由も

□ 7. 注意点・デメリットを伝えているか
     - 「ただし〜には注意」

□ 8. 専門用語に説明を添えているか
     - 初出時に必ず説明

□ 9. 「先を見せる」テクニックがあるか
     - 「後半で説明します」など

□ 10. 1セクションが長すぎないか
      - 目安：200-400字/セクション

□ 11. セクション間のつなぎが自然か
      - 「続いて〜」「次に〜」

□ 12. NGパターンに該当しないか
      - 誇張、煽り、マウント等
```

### まとめチェック（8項目）

```
□ 1. 振り返りがあるか
     - 「というわけで今回は〜を紹介しました」

□ 2. ポイントが2-4個に整理されているか
     - 多すぎると記憶に残らない

□ 3. 価値を再確認しているか
     - 「〜があれば〜ができます」

□ 4. 行動喚起があるか
     - 「是非試してみてください」

□ 5. 定型締めが正確か
     - 一字一句正確に

□ 6. 短くまとまっているか
     - 約300-400字

□ 7. 新しい情報を追加していないか
     - まとめで新情報はNG

□ 8. NGパターンに該当しないか
```

### 全体チェック（10項目）

```
□ 1. トーンが一貫しているか
     - 「ズボラでも」「簡単に」

□ 2. 40-60代に伝わる言葉遣いか
     - 若者言葉を使っていないか

□ 3. 文字数が基準か
     - 3001字以上である（3000字以下はNG）
     - 目安〜4200字。大幅超過する場合は冗長・離脱リスクとして要確認
     - 5-10分相当の読み上げ体感と矛盾していないか

□ 4. 導入:本題:まとめの比率が適切か
     - 約10-15% : 70-75% : 10-15%

□ 5. 一人称が統一されているか
     - 「私」or「わたし」

□ 6. 敬語レベルが統一されているか
     - です/ます調

□ 7. 操作説明のデバイスが統一されているか
     - PC or スマホ

□ 8. 「ズボラでもできる」感が伝わるか
     - 難しそうに見えないか

□ 9. 実際に5-10分で読み上げられるか
     - 声に出して確認

□ 10. ブランドコアに沿っているか
      - 「楽して結果を出す」
```

## 品質スコアリング

### 採点基準

| スコア | 基準 |
|--------|------|
| A（90-100点） | 即使用可能。修正不要 |
| B（70-89点） | 軽微な修正で使用可能 |
| C（50-69点） | 部分的な書き直しが必要 |
| D（50点未満） | 大幅な書き直しが必要 |

### 減点基準

| 項目 | 減点 |
|------|------|
| NGパターン該当（1件につき） | -15点 |
| 導入チェック未達（1項目につき） | -3点 |
| 本題チェック未達（1項目につき） | -2点 |
| まとめチェック未達（1項目につき） | -3点 |
| 全体チェック未達（1項目につき） | -2点 |

### 合格基準

```
・スコア70点以上（B以上）
・NGパターン該当が0件
・導入の必須5項目（1,3,4,5,6）がすべて達成
・定型締めが正確
```
